# REELTOR — Your Reels, Searchable

**Product Requirements Document v3.0 | March 2026 | Local-First MCP Architecture**

---

## 1. Overview

Reeltor is a local-first MCP server + extraction pipeline that turns Instagram Reels into structured, searchable knowledge. It exposes two tools — `search_reels` and `delete_reel` — via the Model Context Protocol, callable by any MCP-compatible AI assistant (Claude Desktop via stdio, ChatGPT via remote HTTP/SSE).

A background ingest service processes Reels (transcription via OpenAI Whisper API, metadata/description capture, LLM-powered structured data extraction) and stores everything in a local SQLite database with vector embeddings. The MCP server reads from this DB and exposes search/retrieval tools.

## 2. Problem Statement

People save 30–50+ Reels per week but revisit fewer than 10%. When they do, they rewatch entire videos to extract one piece of info. Instagram's save folders are manual, unsearchable, and offer no content preview. Saved Reels become a graveyard of forgotten intentions.

Reeltor plugs into whatever AI assistant the user already talks to — no new app to adopt.

## 3. Architecture

Two components share a single local SQLite database:

### 3.1 Ingest Service (Background Server)

A persistent local process listening for incoming Reel URLs (via CLI or HTTP endpoint). Runs the extraction pipeline on each URL immediately upon receipt and writes structured results + embeddings to the local DB. Thorough error handling at every pipeline stage (download, transcription, extraction, embedding, DB write) with clear error states and logging.

### 3.2 MCP Server (Official SDK)

A Model Context Protocol server built with `@modelcontextprotocol/sdk` that reads from the same SQLite file. Supports two transports:

- **stdio** — for Claude Desktop (zero infrastructure, local)
- **Streamable HTTP** — for ChatGPT and other remote clients (exposed via ngrok)

**Two tools only:** `search_reels` and `delete_reel`. This is the primary interface — there is no separate frontend.

## 5. Database Schema

All data lives in a single SQLite file. Vector search via `sqlite-vec`.

### 5.1 `reels` table

| Column | Type | Description |
|--------|------|-------------|
| id | TEXT PK | UUID |
| url | TEXT UNIQUE | Normalized Instagram Reel URL |
| transcript | TEXT | Full transcription from OpenAI Whisper API |
| description | TEXT | Instagram Reel caption/description metadata |
| structured_data | JSON | Type-specific extracted info (recipe steps, locations, product details, etc.) |
| reel_type | TEXT | Detected category: recipe, travel, product, tutorial, music, advice, workout, other |
| summary | TEXT | One-line summary generated by LLM |
| tags | JSON | Auto-generated tags |
| user_message | TEXT | Optional custom message from user at save time |
| collection | TEXT | User-specified or auto-assigned collection name |
| created_at | DATETIME | Timestamp of processing |
| source_duration | REAL | Original Reel duration in seconds |
| status | TEXT | processing, complete, error |
| error_message | TEXT | Error details if status is "error" |

### 5.2 `embeddings` (sqlite-vec virtual table)

| Column | Type | Description |
|--------|------|-------------|
| reel_id | TEXT | References reels.id |
| chunk_text | TEXT | The text chunk that was embedded |
| embedding | FLOAT[1536] | Vector embedding via OpenAI text-embedding-3-small |

Embeddings generated from: transcript chunks, description, structured_data JSON, summary. Each Reel produces 3–8 embedding rows depending on content length.

## 6. Extraction Pipeline

Triggered immediately when the ingest service receives a Reel URL:

1. **Validate & Deduplicate** — normalize URL (strip tracking params, canonicalize `/reel/` vs `/p/` formats), check DB for existing entry. Early exit if duplicate.
2. **Download Video** — yt-dlp downloads Reel video + metadata (description/caption) to temp storage
3. **Extract Audio** — ffmpeg strips audio → send to OpenAI Whisper API (`whisper-1`) → returns transcript
4. **Single Claude Call (Classification + Extraction)** — one Claude API call receives: transcript + description/metadata. Returns:
   - `reel_type` — detected category
   - `summary` — one-line TL;DR
   - `structured_data` — type-specific JSON
   - `tags` — auto-generated tags
   - `collection` — suggested collection
5. **Embedding Generation** — chunk all text content (transcript, description, structured_data, summary), send to OpenAI `text-embedding-3-small`, insert into sqlite-vec
6. **Write to DB** — insert reel record + embedding rows in a single transaction
7. **Cleanup** — delete temp video/audio files

### Error Handling

Every pipeline stage is wrapped with error handling:

| Stage | On Failure |
|-------|-----------|
| URL validation | Reject immediately, return error to caller |
| yt-dlp download | Log error, set reel status to "error" with message, stop pipeline |
| Whisper transcription | Log warning, continue with empty transcript (description may still be useful) |
| Claude extraction | Log error, set status to "error", keep raw transcript/description in DB for retry |
| Embedding generation | Log warning, set status to "complete" (search will fall back to text matching) |
| DB write | Log error, set status to "error" |

### Handling No-Audio Reels

Many Reels are text-on-screen with background music. In these cases:
- Whisper returns minimal/no useful transcript
- The Reel's Instagram **description/caption** becomes the primary signal
- Claude extraction still runs on whatever text is available (description alone is often sufficient)
- This is an acceptable tradeoff for MVP — no OCR/frame extraction

### Extraction Templates by Reel Type

| Reel Type | structured_data format |
|-----------|----------------------|
| Recipe | `{ ingredients: [...], steps: [...], cook_time, servings }` |
| Travel | `{ locations: [{ name, area, tips, cost }] }` |
| Product | `{ product_name, verdict, price, where_to_buy, pros, cons }` |
| Tutorial | `{ steps: [{ step_num, instruction }], tool_required }` |
| Music | `{ song_name, artist, genre, mood }` |
| Advice | `{ takeaways: [...], context }` |
| Workout | `{ exercises: [{ name, sets, reps, rest }], target_muscle, duration }` |
| Other | `{ key_points: [...], entities_mentioned: [...] }` |

## 7. MCP Server — Tool Specifications

Built with `@modelcontextprotocol/sdk`. All tools read from `~/.reeltor/reeltor.db`.

### `search_reels`

Semantic search across all saved Reels. Combines vector similarity with metadata filtering.

```
Input:
  query: string        — Natural language search query (required)
  reel_type: string    — Optional filter by type (recipe, travel, etc.)
  collection: string   — Optional filter by collection
  limit: number        — Max results (default 5)

Output:
  results: [{
    reel_id, url, summary, reel_type, structured_data,
    tags, collection, created_at, relevance_score
  }]
```

### `delete_reel`

Remove a Reel and its associated embeddings.

```
Input:
  reel_id: string

Output:
  deleted: boolean
```

## 8. Ingest Methods (How Reels Enter the System)

| Method | How It Works | MVP? |
|--------|-------------|------|
| CLI | `reeltor add <url> --message "..."` | Yes |
| HTTP endpoint | POST to `localhost:7433/ingest` with `{ url, message }` | Yes |
| iOS Shortcut | Share sheet → Shortcut POSTs Reel URL to ingest endpoint (via ngrok) | Yes |

Reels are processed immediately upon receipt — no queue, no polling. The ingest endpoint returns a reel_id and begins processing synchronously.

## 9. MCP Client Setup

### Claude Desktop (stdio — recommended for development)

Edit `~/Library/Application Support/Claude/claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "reeltor": {
      "command": "node",
      "args": ["/path/to/reeltor/mcp-server.js"]
    }
  }
}
```

Restart Claude Desktop. Tools appear immediately.

### ChatGPT (remote HTTP/SSE — for demo)

1. Start MCP server with HTTP transport: `reeltor serve --http`
2. Expose via ngrok: `ngrok http 7433`
3. In ChatGPT: Settings > Apps & Connectors > Add custom connector > paste ngrok URL
4. Tools appear in ChatGPT (requires Developer Mode on Pro/Plus/Business plans)

### Single Server, Both Transports

The MCP server supports both stdio and Streamable HTTP from the same codebase. No Mastra dependency needed.

## 10. MVP Scope

### In Scope (Phase 1)

1. Local ingest service with HTTP endpoint + CLI
2. Extraction pipeline: yt-dlp → metadata capture → Whisper transcript → single Claude call → embeddings → SQLite
3. SQLite + sqlite-vec local database
4. MCP server via official SDK with 2 tools (`search_reels`, `delete_reel`)
5. Dual transport: stdio (Claude Desktop) + Streamable HTTP (ChatGPT via ngrok)
6. Thorough error handling with status tracking
7. No auth (single user demo)

### Out of Scope (Phase 2+)

- Additional MCP tools (update_reel, list_collections, etc.)
- Telegram/WhatsApp bot ingest
- Frame extraction / OCR / key frames
- Multi-user / authentication
- TikTok and YouTube Shorts support
- Web dashboard / mobile app
- Cloud hosting (beyond ngrok)
- Embedding model migration tooling

## 11. User Flow (MVP)

1. User clones repo, runs `npm install`
2. Runs `reeltor init` → creates `~/.reeltor/`, prompts for OpenAI + Anthropic API keys
3. Runs `reeltor start` → ingest service + MCP server boot up
4. User connects MCP server to Claude Desktop (stdio) or ChatGPT (ngrok + HTTP)
5. User finds an interesting Reel on Instagram
6. Sends URL via CLI (`reeltor add <url>`) or HTTP POST
7. Pipeline runs immediately (10–30s), reel is searchable
8. User asks their AI assistant: "what was that coffee shop reel?"
9. Assistant calls `search_reels` → retrieves structured data
10. Assistant renders a contextual response with extracted info + original URL

## 15. Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Instagram blocks yt-dlp scraping | High | Test early; may need cookies/session auth; monitor for changes |
| No-audio Reels produce thin results | Medium | Description metadata is primary fallback; acceptable for MVP |
| LLM extraction quality varies by type | Medium | Eval suite per type; few-shot examples in prompts |
| sqlite-vec limitations at scale | Low | Fine for personal use (< 10K Reels) |
| ngrok URL changes on restart | Low | Use ngrok fixed subdomain (paid) or re-paste URL in ChatGPT |
